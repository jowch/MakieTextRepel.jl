# Makie `annotation!` — Limitations, TODOs, and API Stability

## Abstract

This document audits Makie 0.24.10's `annotation!` recipe at
`/home/jonathanchen/.julia/packages/Makie/p9K7f/src/basic_recipes/annotation.jl`
(1190 lines) and the supporting infrastructure for integration risk. The
recipe is **non-experimental** (it ships in `basic_recipes.jl` and has docs +
tests) but its label-placement algorithm is **explicitly disclaimed as
unstable across non-breaking releases** (`annotation.jl:168`). The default
solver `LabelRepel` is a fixed-iteration (default 200), O(maxiter · N²)
force-based repeller with three hard-coded tunables (`repel`, `attract`,
`padding`) and **no convergence detection, no jitter, no anisotropy, no
label-drop, no per-label weighting**. The only documented extension point is
multiple-dispatching `Makie.calculate_best_offsets!` on a new algorithm
singleton (`annotation.jl:421`, `:454`) — but the *signature* (positional
`offsets, textpositions, textpositions_offset, text_bbs, bbox` + kwargs
`maxiter`, `labelspace`, and a `reset::Bool` only on the `::Automatic`
overload) is undocumented and not exported. No CHANGELOG / NEWS file ships in
the package, so we cannot trace prior signature changes from this snapshot
alone. Edge-case probes (`examples/probe_annotation_edges.jl`) confirm that
0 labels, empty strings, LaTeX/rich text, log axes, and co-located points
all work; **`rotation` is not a supported attribute** and N=500 fully
re-solves in ~1.5 s on this machine.

## 3. Default `LabelRepel` algorithm — gaps vs ggrepel

### What it is

`annotation.jl:415-419`:

```julia
Base.@kwdef struct LabelRepel
    repel::Float64 = 0.1
    attract::Float64 = 0.1
    padding::Vec2d = Vec2d(6, 5)
end
```

Three knobs only. Padding is in pixels because the entire solver operates in
**pixel space** (offsets are `Vec2f`, bboxes are projected to screen via
`screenpoints_target` / `screenpoints_label`, and the bbox the solver
constrains against is `Rect2d((0,0), widths(viewport))`,
`annotation.jl:311-312`).

### Algorithm body (annotation.jl:454-538)

Loop runs `maxiter = 200` by default (`annotation.jl:460`), with **no early
exit**. Each iteration does, in order:

1. **Pairwise box repulsion** (`:483-491`): O(N²) double loop. Uses
   `rect_overlap` which is anisotropic — it returns the dominant overlap
   direction (the axis on which the boxes overlap less, `:412-419`,
   `rect_overlap` definition at `:553-570`) and pushes both labels
   symmetrically.
2. **Self-attraction to anchor** (`:494-499`): each label is pulled toward
   its own data point with strength `attract = 0.1`. Uses
   `distance_point_outside_rect` (`:367-388`), which returns the vector from
   the rect *to* the point if the point is outside, zero if inside.
3. **Repulsion from all anchor points** (`:502-509`): O(N²) — every label
   bbox checks every anchor point and is pushed away from any anchor it
   contains (uses `distance_point_inside_rect`, `:390-413`). Critically,
   this is the **only** "don't overlap a data point" force, and the
   distance returned by `distance_point_inside_rect` is the *smaller of dx /
   dy* (`:408-412`), so labels can only escape a point along one axis at a
   time.
4. **Viewport clamp** (`:512-528`): hard clip — if `pl + ox < l` the offset
   is replaced with `Vec(l - pl, oy)` etc. This is a per-iteration
   projection onto the viewport (the same hack we replicate in
   `MakieTextRepel`).

### Center-bias kick (annotation.jl:467-477)

If all `offsets` start at zero, the solver kicks each label slightly
toward the viewport center with magnitude `0.1 * repel / norm(v)`. This
breaks symmetry deterministically (no RNG). For co-located labels (`v` is
the *target → center* vector, identical per overlapping pair), the kicks
are identical — and the iteration relies on the subsequent pairwise
`rect_overlap` to actually separate them. Probe 2 confirms 5 labels at
`(1,1)` end up at 5 distinct offsets (5 distinct directions). The
zero-vector special case (`n > 0 ? … : zero`, `:475`) was added to fix the
StackOverflow described at `test/plots/primitives.jl:184-194` and
https://github.com/MakieOrg/Makie.jl/issues/3551.

### Comparison with ggrepel (R, the de-facto state of the art)

### Other gaps not in the ggrepel comparison

- **No symmetry-breaking when boxes coincide perfectly** beyond the center
  kick. If two labels share both anchor and text, the per-pair repulsion at
  `:483-491` cannot resolve them (their bboxes have identical overlap, the
  symmetric push cancels). Probe 2 saw 5 distinct outcomes only because the
  bboxes were slightly different widths — `string.(1:5)` has variable
  lengths once you go beyond single digits. With identical text on identical
  points, separation is not guaranteed.
- **No "labels go on top of nothing" awareness** — the solver knows about
  anchor points only, not arbitrary scatter markers, lines, gridlines, axis
  ticks. A label can land squarely on an axis tick or an unrelated marker.
- **No persistent state between solves**. `reset = !advance` at
  `annotation.jl:317` discards offsets whenever inputs change (other than
  `__advance_optimization`). Animations that move data points slightly will
  re-solve from a center kick every frame, which is the opposite of what
  you usually want for visual stability.
- **No `Inf`/`NaN`-anchor guard**. `_guard_nonfinite` (`:233`) handles
  non-finite bboxes, but if an anchor's *target_positions* contains `NaN`
  the projected `screenpoints_target` will be `NaN`, and
  `distance_point_outside_rect` / `distance_point_inside_rect` will
  propagate it into `offsets`. Not probed here, but the same concern that
  motivated the `_guard_nonfinite` workaround.

### Algorithmic complexity

Per iteration:

```
T(N) ≈ N(N-1)/2  pairwise box-box   (annotation.jl:483-491)
     +  N        attract            (annotation.jl:494-499)
     +  N²       label-bbox × anchor (annotation.jl:502-509)
     +  N        viewport clamp     (annotation.jl:512-528)
     = O(N²)
```

Total: `O(maxiter · N²)`. With `maxiter = 200` and N = 500 → 5 × 10⁷
operations on each full re-solve. Probe 8 measured **1.54 s wall clock for
N=500** (Julia 1.11, CairoMakie 0.15.10, no rendering of the image — mostly
solve time). Extrapolating naively: N=1000 → ≈ 6 s, N=2000 → ≈ 24 s.

Bottlenecks for >500 labels:

- The double loop at `:483-491` allocates nothing (uses `offset_bbs` in
  place) but is 6 reads / 4 writes per pair. No SIMD-ization.
- `rect_overlap` (`:553-570`) and `interval_overlap` (`:540-552`) are
  branch-heavy; not obviously inlinable.
- The label-vs-all-anchors loop (`:502-509`) **always traverses all
  anchors, even ones nowhere near the label**. A spatial index (grid hash,
  kd-tree) would cut this to O(N log N) or better, but Makie doesn't ship
  one for screen-space points.
- No batching across `maxiter`. Each iteration also re-allocates the
  broadcast `offset_bbs .= padded_bbs .+ offsets` (`:480`) — but in-place
  via `.=` so likely zero-alloc after warmup; not measured.

## 4. Coordinate-frame caveats (`labelspace`)

### Where it branches

```
annotation.jl:163   labelspace = :relative_pixel    (default)
annotation.jl:269   :labelspace is an input to the screenpoints_label computation
annotation.jl:275-287   branch on (space === :relative_pixel) vs else
annotation.jl:296-297   labelspace also threaded into the offset computation
annotation.jl:313    passed to calculate_best_offsets!
annotation.jl:425, :457   labelspace::Symbol param threaded down — but the
                          LabelRepel solver never reads it.
```

So `labelspace` is **threaded through to the solver but the solver does not
use it**: `calculate_best_offsets!(algorithm::LabelRepel, …)` (`:454-538`)
takes the kwarg, then ignores it. The solver always works in pixel space.
The only place `labelspace` actually changes behavior is the
`screenpoints_label` computation at `:275-287`:

```julia
if space === :relative_pixel
    return (tps .+ loffpos,)    # treat loffpos as a Vec2 in pixels
else
    # :data — project loffpos through transform_func + f32c + model
    transformed_label_pos = apply_transform(tf, loffpos)
    f32c_mat = f32_convert_matrix(f32c)
    return (_project(Point2f, proj * f32c_mat * model, transformed_label_pos),)
end
```

This means `:data` triggers a full data-to-pixel reprojection that depends
on `world_to_pixel`, `f32c`, `model`, `transform_func`, while
`:relative_pixel` short-circuits and skips updates from camera/transform-
func changes (`annotation.jl:280-282`).

### Behavioral consequences

- **Auto-placement implies `:relative_pixel`** in spirit. When you call
  `annotation!(ax, xs, ys; text)` without explicit label offsets, the
  `Vec2d(NaN)` placeholder (`:197, :201, :213, :222`) gets handed to the
  `screenpoints_label` branch and would project to NaN under `:data`. With
  the default `:relative_pixel`, `NaN .+ pixel_target` is still NaN — but
  then the solver overwrites `offsets` (`:316` `resize!(offsets, …)`) and
  the rendered offset is the solver's, not the NaN. Switching to
  `labelspace = :data` while still using auto-placement is poorly defined
  and not exercised in the docs.

- **Mixed labelspace is impossible per-label.** `labelspace` is a scalar
  attribute, not a per-label vector. If a downstream package wants some
  labels in data space (e.g., a fixed annotation A→B) and others auto-
  placed in pixel space, two separate `annotation!` calls are required.

- **The `:data` branch is sensitive to camera updates**. The
  `:relative_pixel` branch deliberately skips updates from `world_to_pixel`
  and `transform_func` (`:280-282`); `:data` does not, so panning/zooming
  re-fires the projection. Performance implication: animated cameras with
  `:data` labelspace pay the projection cost every frame.

- **Log axes work** (probe 6): coordinates pass through
  `apply_transform(tf, loffpos)`. But the solver still operates in
  *uniform* pixel space, so the visual "repel distance" is a constant
  number of pixels regardless of where on the log axis the label lives.
  This is usually what you want, but it means "the closest log-scale data
  cluster gets the most cramped labels" is a real failure mode.

- **3D / Axis3 — unverified**. The code is 2D throughout
  (`Vec2d`/`Point2`/`Rect2d`); applying `apply_transform_and_model` to a
  `Rect3f` in `boundingbox(p::Annotation, …)` (`:592-593`) is the only 3D
  contact. No 3D test exists. Likely renders, likely buggy at oblique
  camera angles.

## 5. Edge cases (verified by probe)

The probes live at `examples/probe_annotation_edges.jl` and were executed
with Julia 1.11, Makie 0.24.10, CairoMakie 0.15.10. All probes succeeded
unless noted.

### (a) Zero labels — Probe 1

```julia
annotation!(ax, Float64[], Float64[]; text = String[])
# OK: 0 labels -> offsets length = 0
```

Works. `convert_arguments` at `annotation.jl:222-225` returns
`fill(Vec2d(NaN), 0)` and `Point2d[]`, the solver's length check at
`:427-432` short-circuits when the lengths are all zero (well-formed but
empty).

### (b) All labels at the same point — Probe 2

```julia
xs = fill(1.0, 5); ys = fill(1.0, 5)
annotation!(ax, xs, ys; text = ["A","B","C","D","E"])
# offsets = 5 distinct Vec2f, no NaN
```

Works because:
1. The center kick at `annotation.jl:467-477` produces nonzero `v` for
   non-center anchors (here all 5 anchors map to the same `(1,1)`-projected
   pixel, but `bb_center` differs per label only via the *text* width — so
   each `bb_center → center` vector is identical; the kicks **are
   identical**).
2. The pairwise repulsion at `:483-491` then separates the now-overlapping
   bboxes asymmetrically because `rect_overlap` returns equal-magnitude
   opposing vectors that sum to zero only for the symmetric pair; running
   200 iterations of cascading pairwise pushes resolves them into the
   compass-rose pattern shown.

Failure mode: if all 5 labels were *identical text*, the bboxes would be
identical, the kick would be identical, and the pairwise overlap would be
maximally symmetric. In practice the 200 iterations amplify
floating-point asymmetries enough to converge — but there is no formal
guarantee. Not deterministic in the "if I run the same code on a different
machine I get the same answer" sense, although it is RNG-free.

### (c) Empty strings — Probe 3

```julia
annotation!(ax, [1,2,3], [1,2,3]; text = ["A", "", "C"])
# offsets = Vec2f[[10.67, …], [0.0, 5.0], [-11.05, …]], no NaN
```

Works because `_guard_nonfinite` (`annotation.jl:233`) replaces the empty
string's non-finite `text!` bbox with `Rect2d(0, 0, 0, 0)`. The fix is
recent — see https://github.com/MakieOrg/Makie.jl/issues/3551 referenced at
`test/plots/primitives.jl:75, :184-194`. Pre-fix, this triggered an
infinite recursion in ComputePipeline because `normalize(zero_vector)`
produced NaN offsets that defeated the convergence check
(`primitives.jl:185-189`).

### (d) LaTeX / rich text — Probes 4, 5

```julia
annotation!(ax, [1,2], [1,2]; text = [L"\alpha^2", L"\sum_{i=1}^n x_i"])
annotation!(ax, [1,2], [1,2]; text = [rich("bold ", rich("italic"; font=:italic)), rich("normal")])
```

Both succeed. `annotation!` forwards `text` directly into the child
`text!` plot at `annotation.jl:238-249`, so any text-engine input that
`text!` accepts works. The bbox tracking goes through the same
`register_raw_string_boundingboxes!` path
(`/home/jonathanchen/.julia/packages/Makie/p9K7f/src/basic_recipes/text.jl:586`),
which handles `LaTeXString` and `RichText` via `MathTeXEngine`.

Caveat: the bbox returned for LaTeX strings tracks the **ink** bounds of
the MathTeX layout, not the typesetter's logical bounding box. Tight
clusters of LaTeX labels can therefore overlap descenders / subscripts
without the solver noticing (similar to the empty-string case but more
subtle).

### (e) Text rotation — Probe 10

```julia
annotation!(ax, [1,2], [1,2]; text = ["A","B"], rotation = pi/4)
# Makie.InvalidAttributeError: Invalid attribute rotation for plot type Annotation{…}
```

**`rotation` is not a supported attribute.** The available attributes
returned by the error are:

```
algorithm, align, clipstart, color, font, fonts, fontsize,
justification, labelspace, lineheight, linewidth, maxiter, path,
shrink, style, text, textcolor, visible
```

This is consistent with the `@recipe Annotation` block at
`annotation.jl:95-173` — `rotation` is not listed. The annotation's text
is always horizontal in pixel space. The only rotation that exists inside
the recipe is **arrowhead** rotation (`annotation.jl:1030-1099`,
`head_rotation`, `tail_rotation`), computed from the connector tangent.

This is a significant limitation if a downstream package wants to lay out
diagonal labels for dense scatter plots.

### (f) Log / non-linear scaling — Probe 6

```julia
ax = Axis(fig[1,1]; xscale = log10, yscale = log10)
annotation!(ax, [1,10,100,1000], [1,10,100,1000]; text = ["a","b","c","d"])
# OK
```

Works. The `screenpoints_target` computation goes through the same
`register_projected_positions!`
(`/home/jonathanchen/.julia/packages/Makie/p9K7f/src/utilities/projection_utils.jl:49`)
that all Makie plots use, which honors `transform_func`. The solver then
sees uniformly-spaced pixel positions and produces the same pixel-domain
offsets it would for linear axes. Visually, the offsets are *constant in
pixels* even though they correspond to wildly different *log-decade*
fractions — this is the standard convention and matches `:relative_pixel`.

### (g) Labels larger than viewport — Probe 7

```julia
fig = Figure(size = (200, 200))
annotation!(ax, [0.1,0.5,0.9], [0.5,0.5,0.5];
            text = ["a-really-long-label-A", "long-B-label", "long-C-label"])
# offsets land at [54.62, -24.12], [-26.65, 15.35], [-29.73, -8.77]
```

Works but the viewport clamp at `annotation.jl:512-528` is doing a lot of
work. The first label's offset of +54 px x in a 200 px viewport is large
enough that the clamp is the only thing keeping the label visible. With
labels that are *individually* wider than the viewport, the clamp will pin
both `pl + ox < l` and `pr + ox > r`, with the last assignment winning —
i.e., the right-edge clamp will overwrite the left-edge clamp and the
label will hang off the *left* edge. This is the same brittleness our
own `compute_offsets_clamped!` patched out by clamping x and y
independently (which it actually does too — `annotation.jl:514-526` uses
separate `if/elseif` branches per axis, and the second branch overwrites
the first in the impossible case).

### Other edge cases not probed

- **`Inf` or `NaN` in target positions** — `_guard_nonfinite` guards the
  bbox path but not the anchor projection. Untested; risk of `NaN`
  propagating into `offsets`.
- **Labels with `visible = false`** — visibility is read once at
  `annotation.jl:356` (`plotlist!(p, p.plotspecs; visible = p.visible[])`).
  TODO at `:355` says dynamic toggling doesn't propagate. The label text
  is in a separate `text!` plot (`:238-249`) and *does* respect dynamic
  `visible`. Result: toggling `visible` hides the *connectors* one way
  (the first `[]` value) but not dynamically.
- **`maxiter = 0`** — Probe 9 confirms this is accepted; the loop at
  `:479` simply doesn't run, but the center kick at `:467-477` still
  fires. Offsets are non-zero (the kick scaled by `0.01`).
- **`maxiter` < 0** — not probed. The loop `for _ in 1:maxiter` is a no-
  op for negative ints, so behaviorally the same as `0`. No error.

## 6. Interactivity gaps

There is **no built-in drag / hover / manual-nudge support** in the
recipe.

- The recipe registers exactly one Observable bridge:
  `on(offsets -> update!(txt, offset = offsets), p.offsets, update = true)`
  at `annotation.jl:326`. This is one-way (solver writes, text reads).
- The `offsets` ComputePipeline node (`annotation.jl:298-321`) is owned by
  the solver. If a user sets `p.offsets[] = something`, that something
  will be overwritten on the next input change because `reset = !advance`
  (`:317`) discards the current offsets whenever any non-
  `__advance_optimization` input fires. To "lock" a manually nudged
  position you would need to either set `maxiter = 0` and then write
  `offsets`, or wrap the algorithm.
- `advance_optimization!(p, n)` (`annotation.jl:361-365`) is the only
  programmatic step-control. It sets `p.__advance_optimization = n` which
  triggers an `advance = true` re-solve that **keeps current offsets** as
  the starting point (because `reset = !advance`). Useful for animation /
  warm-start, but not exported (`__advance_optimization` is double-
  underscored — the canonical Julia convention for "really really
  internal").
- No `pickable`/`inspectable` interaction is registered for the label
  bboxes. A user can click on the underlying `text!` plot (because Makie's
  text plot is generally pickable), but there's no hook from the label
  click back to "nudge this offset."
- Per-label motion locking, mentioned at TODO `annotation.jl:447-449`, is
  not implemented.

For a downstream package wanting "click and drag the label," the path is
roughly: subscribe to `p.offsets`, register a mouse handler on the parent
scene, write to `p.offsets[i] = …`, and use `maxiter = 0` (or a custom
algorithm) to prevent the solver from overwriting on the next tick. There
is no API affordance to make this clean.

## 7. Integration risk for downstream packages extending `calculate_best_offsets!`

The natural extension point is multiple-dispatch on the algorithm type:

```julia
struct MyAlgorithm
    # ...
end
function Makie.calculate_best_offsets!(
        ::MyAlgorithm, offsets, textpositions, textpositions_offset, text_bbs, bbox;
        maxiter, labelspace,
    )
    # custom solver
end
```

A user opts in by setting `algorithm = MyAlgorithm()` on the plot. The
contract that downstream packages depend on:

### What can change in a non-breaking Makie release

The docstring says positioning may change. In practice the broader API
surface that affects downstream extensions is:

1. **The signature of `calculate_best_offsets!`** — there are two
   overloads (`::Automatic` at `annotation.jl:421`, `::LabelRepel` at
   `:454`). The `::Automatic` overload takes an extra `reset::Bool` kwarg
   that the `::LabelRepel` overload does not. The recipe calls the
   `::Automatic` form via `calculate_best_offsets!(args.algorithm, …)` at
   `:306`, so a user-defined algorithm must match the `reset`-receiving
   signature — i.e., the kwargs are `maxiter`, `labelspace`, and
   `reset`. **The `reset` kwarg is undocumented**. Future versions could
   add more kwargs without violating semver (because the signature isn't
   public). Already, there's a discrepancy between the two overloads
   shipped in the same file.

2. **The semantics of the `reset` flag**. Currently it means "if false,
   keep the previous offsets, run `maxiter` more steps; if true, treat
   offsets as a fresh start." A downstream impl that ignores this will
   silently break animation/warm-start.

3. **`labelspace` is passed but not used by `LabelRepel`**. A future
   release could meaningfully use it (e.g., to support data-space
   repulsion), changing the operational semantics for any downstream
   algorithm that accepts the kwarg and discards it.

4. **The bbox is `Rect2d((0, 0), widths(viewport))` in pixel coordinates**
   today (`annotation.jl:311-312`). If a future release passes a non-
   pixel-space bbox under some conditions (e.g., `:data` labelspace), a
   downstream algorithm computing pixel offsets would be wrong.

5. **`text_bbs` and `textpositions` are pre-projected** to pixel space
   (`screenpoints_target` and `screenpoints_label`, see `:295-310`). This
   is true today regardless of `labelspace`. A future release that
   changes `screenpoints_label` semantics (currently the data-space
   branch projects through `tf` and `f32c`, `:283-285`) would shift the
   meaning of the inputs.

6. **The `offsets` output is `Vec2f` (pixel space)**. The `text!` plot
   consumes them as a screen-space `offset`. If Makie ever moves the
   `offset` semantics of `text!` to data space, the solver output
   becomes wrong.

7. **The struct `LabelRepel` is not exported and not `@compat public`**.
   Renaming or splitting it (e.g., into `LabelRepel` + `LabelRepelV2`) is
   non-breaking for Makie but would silently break any downstream that
   pattern-matched on the type.

8. **The `Ann.*` baremodules** (`annotation.jl:2-89`) are also not
   exported and not declared public. `Ann.Paths.Line`, `Ann.Paths.Corner`,
   `Ann.Paths.Arc`, `Ann.Arrows.Line`, `Ann.Arrows.Head`,
   `Ann.Styles.Line`, `Ann.Styles.LineArrow`, `Ann.Styles.WithText` —
   all internal symbols by Julia convention.

9. **The `p.offsets` Observable** is publicly mentioned in the test file
   (`test/plots/primitives.jl:161-163, :193`) but not documented in
   `annotation.jl`. It's the most reasonable subscription point for a
   downstream "watch the solver" workflow.

10. **`advance_optimization!`** is *defined as a function* in the source
    but never exported. Calling `Makie.advance_optimization!(p, 5)` works
    today; nothing prevents Makie from renaming it.

11. **`__advance_optimization`** is a hidden ComputePipeline input
    (`:290`). The double-underscore is the strongest "do not touch" signal
    Julia has. The test file uses it directly (`primitives.jl:162`) — so
    Makie's own tests pin the name, but there's no contract.

### Git tags / CHANGELOG

This snapshot has **no `CHANGELOG.md`, no `NEWS.md`**, and no `docs/`
directory inside the installed package (we are at `Makie/p9K7f`, version
`0.24.10`). The annotation recipe was added in a single file commit (no
prior versions of `basic_recipes/annotation.jl` are in this tarball to
diff against), so we cannot trace historic signature changes from the
installed package alone. The GitHub history would be required for that —
not done here per scope.

Recommendation for downstream integration risk mitigation:

- **Pin a Makie minor** (`Makie = "0.24"`) and treat 0.25 as a breaking
  bump for the purposes of label algorithms even if it's
  semver-non-breaking by Makie's standard, because the
  `calculate_best_offsets!` signature is undocumented.
- **Define the user's algorithm dispatching on a `<:` of a sentinel
  abstract** that the downstream package owns, so adding a new positional
  arg in Makie surfaces as a `MethodError` rather than silently calling
  the wrong overload.
- **Re-implement the `reset` semantics**: accept both `reset=true` and
  the default. Pin the warm-start behavior in tests against your own
  fixture.
- **Subscribe to `p.offsets`** rather than reading `p.__advance_optimization`.
- **Add an integration test** that catches changes in
  `Makie.LabelRepel`/`Makie.calculate_best_offsets!` method signatures
  (using `methods(Makie.calculate_best_offsets!)` introspection on
  Makie upgrade).

## 8. ComputePipeline coupling

The recipe is built on the newer ComputePipeline (`register_computation!`,
`register_camera_matrix!`, `add_input!`, `add_constant!`, `map!(p, ...)`)
rather than the legacy Observables-everywhere style. This has integration
consequences:

- `register_computation!` blocks must return `(nothing,)` to skip output
  updates (see `annotation.jl:281-282, 284`). Mis-returning is silent and
  causes downstream nodes not to fire.
- The `changed` named tuple passed into the callback is used at
  `:300` (`sum(values(changed)) == 1 && changed.__advance_optimization`).
  Replicating this style in a downstream algorithm is fiddly because the
  field names are positional (`:algorithm`, `:screenpoints_target`, …
  per `:295-297`).
- `:label_offsets_or_positions` is the canonical name of arg1
  (`annotation.jl:95, 269, 282`). Renaming the first positional argument
  in a recipe call to update offsets means writing
  `update!(p, arg1 = …)` (see test at `primitives.jl:166, 170, 175, 179`),
  not `update!(p, label_offsets_or_positions = …)`. Inconsistent but
  intentional.
- The `:offsets` output node and the `p.offsets` Observable are linked
  via the explicit `on(offsets -> update!(txt, offset = offsets), …)`
  bridge at `:326`. This synchronization is **manual** because text's
  `offset` attribute is sized at construction (`zeros(Vec2f,
  length(p.target_positions[]))`, `:243`) and ComputePipeline does not
  automatically reconcile sizing across stages — hence the explicit
  `resize!(offsets, length(args.screenpoints_target))` at `:303`.

## 10. Top integration concerns for `MakieTextRepel`

A condensed list for downstream design discussions:

1. **Algorithm signature is undocumented.** Pin a Makie minor and add a
   method-introspection test against `Makie.calculate_best_offsets!`.
2. **`reset::Bool` kwarg only on the `::Automatic` overload** is the most
   subtle gotcha. Downstream `::MyAlgorithm` methods must accept it.
3. **No convergence detection** — downstream solvers can simply add one;
   not a *break* risk but a "better-than-Makie" opportunity.
4. **`p.offsets` is the only stable handoff point** but isn't formally
   documented. Tests use it (`primitives.jl:161-193`).
5. **`Ann.Paths.*` / `Ann.Styles.*` are not exported.** Re-exporting them
   from a downstream package risks a name collision if Makie ever promotes
   them.
6. **No `rotation` attr.** Diagonal labels need a parallel approach
   (custom `text!` with `rotation`, or a different recipe).
7. **The viewport clamp is per-iteration and per-axis** (`:512-528`). It
   *can* push labels off-viewport on the conflicting axis if both x and y
   constraints fire. We already work around this in `MakieTextRepel`.
8. **`reset = !advance` discards offsets on every input change** other
   than `__advance_optimization`. Animation will re-solve from scratch
   each frame unless the package forces `advance` mode — and that means
   touching the `__double_underscored` API.
9. **No `label-drop` / `min-overlap` reporting.** Downstream packages may
   want to expose "labels couldn't fit" diagnostics; this would have to
   be computed externally.
10. **Performance ceiling ≈ N=500 in real time.** For N>1000 a downstream
    package should ship a spatial-index variant.
